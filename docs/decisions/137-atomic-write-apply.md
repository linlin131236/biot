# M137 Decision: 写入路径使用同目录临时文件 + os.replace

## 决策

对批准后的文本写入使用同目录临时文件，然后通过 `os.replace()` 原子替换目标文件。

## 原因

裸 `Path.write_text()` 会直接打开目标文件写入。若进程崩溃、磁盘错误或写入中断，目标文件可能处于部分写入状态。批准后的补丁应用属于高风险写入路径，应该保证失败时尽量不破坏原文件。

同目录临时文件的优点：

- 与目标文件处于同一文件系统，`os.replace()` 可保持原子替换语义。
- 替换失败时原目标仍保留。
- 代码改动小，便于验证。

## 取舍

- 不引入复杂事务系统。
- 不做备份目录或多版本保存；checkpoint/版本控制仍是更高层恢复能力。
- Windows 上 `os.replace()` 行为满足当前本地桌面应用需求。

## 风险

- 如果临时文件写入成功但替换失败，临时文件会尝试清理；清理失败时可能留下 `.tmp` 文件。
- 原子写只覆盖文本写入路径，不代表所有工具执行都具备事务性。
