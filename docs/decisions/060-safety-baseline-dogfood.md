# M60 决策：Safety Baseline Dogfood

## 决策
M60 不新增后端代码或 API，而是对 M55-M59 的 5 个安全子系统 + PermissionGate 边界做全面复盘。决策为"安全底座验收通过"，允许项目进入 V2 Agent 工作流。

## 关键决策点

### 1. M60 作为纯审查 milestone
M60 的交付物是 review gate 文档而非代码。理由：
- M55-M59 每个都有独立的后端/前端/测试交付
- M60 的角色是审查这 5 个子系统之间的协作安全
- 新增代码会引入新的测试负担和风险，违背"复盘"初衷

### 2. 安全红线扫描策略
逐项扫描而非采样：
- `as any` / `unknown as`：全量代码扫描
- Renderer 暴露：全量前端扫描
- Subprocess 白名单：逐文件验证
- 自动执行路径：全量 API 审查
- PermissionGate 边界：逐接口检查

结论：所有红线通过，无新增违规。

### 3. P1/P2 状态确认
M57 复审发现的 P1（脱敏占位符误判）和 P2（硬编码 milestone）已在 `e9a7480` 修复。M60 再次确认：
- Secret scan 排除已脱敏占位符
- milestone 解析为动态正则匹配
- 无回退

### 4. 进入 M61 的前置条件
M60 review gate 明确以下条件全部满足：
1. 审计文件损坏可诊断且不崩溃
2. 9 种高风险模式自动脱敏，已脱敏不误报
3. 发布前检查为只读，无自动 push/release
4. 故障恢复策略覆盖 10 个场景，纯人工步骤
5. PermissionGate 边界完整
6. 安全红线全部重新扫描通过

## 后续风险提示
- M61 Planner Task Graph 是纯内存模型，服务重启后数据丢失。M62+ 引入状态机和持久化前需评估。
- M62+ 引入真实执行（Agent Loop 连接）后，必须重新审计自动执行路径和 PermissionGate 边界。
- 每个新 milestone 都应保持安全红线扫描习惯，不能因为 M60 通过就放松。

## 不做的
- 不在 M60 新增后端服务或 API
- 不修改现有安全子系统
- 不在复盘阶段引入新的执行路径
